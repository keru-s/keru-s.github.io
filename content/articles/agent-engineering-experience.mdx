# Agent 工程经验

## 经验 1：MCP 工具与 Skill 的优先级问题

### 背景

在实际对话中，工具列表里同时存在大量 `mcp__coop__*` 工具和 `coop-skill`。当用户提出“创建需求”这类任务时，模型会优先看到可以直接执行的 MCP 工具，例如 `mcp__coop__create_workitem`，于是下意识跳过 skill，直接调用工具。

问题不在于 skill 本身写得不清楚，而在于对话一开始没有主动加载 `coop-skill`，也没有先读取其中的 `user_config` 和流程约束，导致后续整个创建与状态流转过程都缺少 skill 的指导。对于模型来说，Skill 更像“最佳实践指南”，而 MCP 工具则是“立即可调用的能力”；当两者并存时，如果没有强约束，模型天然倾向于直接动手调用工具。

### 解决方案

#### 方案一：增加提示词描述

核心思路是：**让 skill 在描述层面具备更强的拦截能力，明确要求在调用 MCP 工具前必须先加载 skill。**

具体做法：

- 将 skill 的 description 从普通功能介绍，改为带有强制约束意味的拦截语句
- 以 `MANDATORY:` 开头，给模型一个强信号，表明这不是建议，而是必须遵守的规则
- 在 description 中直接点名 `mcp__coop__*` 工具，让模型在看到这些工具名时，能联想到必须先走 skill
- 增加明确兜底约束，例如“**严格禁止**在未加载此 skill 的情况下直接调用 `mcp__coop__` 工具”，形成闭环

#### 方案二：将 MCP 封装为 Python 脚本调用

后面发现其实还有另外一种方案，就是不要直接启用 MCP，而是将 MCP 封装成 HTTP 的 Python 脚本调用，这样就能有效避免 MCP 占用太多上下文，以及避免 MCP 对 Agent 的干扰。

### 其他可能的方案

- 在 system prompt 或 agent routing 层增加统一规则，规定某类 MCP 工具调用前必须先经过对应 skill
- 对 MCP 工具进行分组或隐藏，降低它们在初始工具列表中的直接暴露度
- 在 skill 加载后由 skill 主动输出操作流程摘要，进一步强化“先读 skill 再执行”的行为模式
- 给 agent 增加工具调用前的检查机制：当命中 `mcp__coop__*` 时，先校验本轮是否已加载相关 skill
- 用更结构化的元数据描述 skill 与 MCP 的依赖关系，而不只依赖自然语言 description

## 经验 2：模型注意力涣散与提示词长度

### 背景

千问三模型在接收超过 1000 token 的复杂嵌套 Markdown 规则时，注意力开始涣散，即使规则中明确写明了执行要求，模型也无法很好地执行任务。重复提醒可能有效，但会导致 system prompt 爆炸，进一步加剧涣散。

### 解决方案

- 使用短、平、快的提示词
- 辅以输入示例作为参考，控制模型行为
- 避免过长的规则嵌套

### 其他可能的方案

- 将长规则拆分为多轮对话或分步指令
- 使用更结构化的输入格式（如 JSON Schema）而非纯文本规则
- 在模型调用前做预校验，确保 prompt 长度在模型高注意力区间内
- 针对不同模型定制不同的 prompt 策略

## 经验 3：JSON 输出解析的工程处理

### 背景

模型输出 JSON 结果时，会在 JSON 前后附带 Markdown 包装，导致 ODPS 数据解析提取失败。有同事采用后训练方式，计算生成比例作为指标推进，但这是 XY 问题的复杂解法。

### 解决方案

在 ODPS 中定义 Python UDF（用户自定义函数），用脚本清理 Markdown 标记，得到平衡的对称字符串（花括号数量对应）。通过 UDF 过滤，JSON 提取率达到 99% 以上。

### 其他可能的方案

- 在后处理层统一做 Markdown 清理（如正则替换）
- 在模型侧通过 fine-tuning 强制纯 JSON 输出
- 在调用处设置 output schema 约束（如 JSON mode）
- 采用更鲁棒的解析器（如能自动剥离 Markdown 包装的库）

## 经验 4：减少 RT（响应时间）的图片与提示词优化

### 背景

在线服务需要根据图片和类目生成商品标题，对 RT 有严格要求。初始链路中图片体积大、提示词冗长，影响模型下载与推理速度。

### 解决方案

- 使用分辨率更小的图片：通过 CDN 后缀压缩，让图片以更低分辨率 / 质量被模型加载，体积从几十 K 降到十几 K，显著降低下载与识别耗时
- 压缩提示词：将长提示词缩减成短提示词，并从中文切换为英文，实验证明能显著降低 RT

### 其他可能的方案

- 使用 WebP / AVIF 等更高效的图片格式
- 在上游做图片预裁剪 / 缩放，避免在模型侧重复处理
- 采用缓存策略（如图片特征缓存、提示词模板缓存）
- 异步化图片预处理，并行化流水线

## 调用固化到 CLI 中

### 背景

今天写了一个需求管理的 skill，最开始是使用 MCP 调用的，整体还可以，但有个问题，由于这个 MCP 提供了五十几个接口，占用了挺多上下文，导致 token 消耗速度比较快。后面发现有人将 MCP 转换成了基础的流式调用，通过调用脚本就可以获取 MCP 返回的结果。

这种方式能有效避免 MCP 的过度加载，但直接使用脚本调用的方式有个问题，就是遇到某些情况的时候，由于返回的 JSON 过程被终端截断了，因此导致 Agent 需要重复调用，并将它写到本地内容中，再写脚本进行解析，大大降低了处理速度。

### 解决方案

后面我思考是否可以通过定义一个 CLI，来将请求 Agent 的调用以及返回结果的解析都固定下来，这样就避免了 Agent 需要编写很长的调用参数，以及自己去处理一些比较麻烦的 JSON 返回。然后我就让 Agent 进行编写，最后编写出来效果特别好。使用了 CLI 这种方式之后，整个调用都变得非常简单，返回也特别简洁，速度也变快了，也能较好地节省 token。
