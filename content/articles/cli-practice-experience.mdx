# CLI 实践经验

## 基础原则

最近在尝试将商品管理的相关接口变成 Agent 使用的 CLI 命令行工具。在这个过程中，由于我是和一个校招生合作的，所以能看到很多一般人在实现接口迁移到命令行工具时的一些错误认识。

首先，好的命令行功能设计绝对不是将原有接口封装和等效重现。原有接口会受到功能迭代的影响，所以会导致设计冗余或者返回复杂，其实并不一定适合 Agent 使用。而是应该封装更多复杂度，在命令行工具中设计更短的参数，并且整个命令、子命令参数都应该更加语义化、更加常见，方便 Agent 使用。

并且设计结果返回时，一定要考虑这种显示是不是适合 AI Agent。很多适合人类的格式并不一定适合 Agent。比如给返回值设计颜色，其实没有必要。比如用表格展示结果，这也不适合 AI 去理解。

还有就是某些接口可能只是返回一个下载链接，然后让人去点击触发浏览器自动下载。那这个时候就要多做一步，直接把这个文件下载回本地。这样能避免 Agent 花很多功夫重新下载到本地，而且下一步的时候也不需要再去找这个文件，能明显减少 token 消耗。

简单来说，设计一个好的命令行工具，就是要把**语义化、避免简单复现、能用代码不用 Agent**作为很核心的设计原则来遵守。

## 方便 Agent 调用的经验

### help 命令很关键

在实现每一个子命令的时候，都务必要让 AI coding 把 `help` 函数完善清楚，说明它的用法以及它的功能。这样能够让 AI Agent 在使用命令行工具时，很方便地去探索所有功能，了解用法，遇到问题的时候也能更快解决，避免为了知道怎么使用而去读源码。

### 增加全局的 Debug 函数

涉及到外部调用的时候，可以考虑在全局增加 Debug 参数。当触发这个参数的时候，就自动将请求的入参和响应都打印出来，方便 Agent 遇到调用问题时排查接口请求和响应是否正确。

如果没有这个参数，那想要确认究竟是入参传错了，还是返回值不符合预期，就会变得尤为困难。

### 涉及网络上传下载时，增加控制参数

当涉及到比较大文件的上传和下载的时候，可以适当考虑在命令中增加一些控制参数，比如等待时长、重试次数等。允许 Agent 在失败重试的时候调大参数，避免当用户网络环境不好的时候频繁失败。

## 开发体会

### 使用 UV

UV 是 Python 开发中最新的版本管理、环境管理工具。由于它是用 Rust 重写过的，因此整体速度非常快。在开发过程中，我们使用它来对依赖版本进行锁定，方便每个研发使用同一套环境来开发命令行工具。

并且由于它支持直接运行某个子命令，因此在修改完函数之后，直接通过 `uv run` 加上命令行子命令，就可以方便地测试当前的子命令是否成功。
