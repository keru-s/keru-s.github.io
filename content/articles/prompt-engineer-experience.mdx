# Prompt Engineer 经验

## 经验 1：选择偏见与位置效应

### 背景

在让大模型进行比较时，如果一直将结果放在某一边（固定位置），大模型会产生位置偏见。这可能导致之前的 ABA 问题未被发现，因为偏见在暗处影响判断。

### 解决方案

- 随机打乱比较项的位置，避免固定位置导致偏见
- 让多个大模型同时进行评判，只有结果一致时才认为有效，避免单个偏见影响结果

### 其他可能的方案

- 引入 blind evaluation（盲评）机制，模型不知道对应关系
- 使用投票或集成方法（多数表决）提升鲁棒性
- 在 prompt 中明确指示忽略位置差异

## 经验 2：双语标题生产的传声筒效应

### 背景

东南亚国家需要双语标题（英文 + 本地语）。如果先本地语→英文→再英文→本地语，会出现传声筒效应：第一次翻译时已产生语义偏差，第二次翻译偏差放大，导致最终产物与原意不符。

### 解决方案

- 先用英文提取核心词汇 / 概念（英文训练质量高）
- 再从英文翻译到本地语
- 如果已有本地语标题，直接采用，避免自行翻译

### 其他可能的方案

- 多语言模型直接本地语→本地语（zero-shot 生成）
- 使用对齐更好的 multilingual 模型做直接生成
- 引入人工校对环节作为最后一道防线

## 经验 3：不同模型的提示词风格差异

### 背景

不同模型对提示词的敏感度差异巨大：

- 千问三：复杂嵌套、深度 Markdown 规则超过 1000 token 时注意力涣散，即使规则清晰也难以执行
- Gemini：只需按顺序组织好要求，无需示例即可自主完成任务

### 解决方案

- 千问三使用短、平、快的提示词，并辅以输入示例控制行为
- Gemini 则可采用完整、结构化的要求列表
- 避免“一刀切”的 prompt 策略，针对不同模型定制

### 其他可能的方案

- 建立模型特性矩阵，自动化生成适配各模型的 prompt 模板
- 使用统一的中间表示（如 JSON Schema）再转为各模型偏好格式
- 在推理前做 prompt 长度与复杂度检测，自动切换策略

## 经验 4：JSON 输出中的 Markdown 包装处理

### 背景

模型在输出 JSON 时经常附带 Markdown 标记（如 `json code fence` 包装），导致下游解析失败（如 ODPS）。有同事采用后训练加比例指标，实为 XY 问题的复杂解法。

### 解决方案

在 ODPS 层定义 Python UDF，脚本清理 Markdown 标记，得到平衡的对称字符串（花括号匹配），提取率达 99%+。

### 其他可能的方案

- 在后处理统一用正则清理 Markdown 包装
- 调用时启用 JSON mode 约束输出格式
- 通过轻量 fine-tuning 使模型习惯纯 JSON 输出
- 使用 JSON 流式解析器自动跳过非 JSON 内容
