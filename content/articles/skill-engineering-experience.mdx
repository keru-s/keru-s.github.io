# Skill 工程经验

## Skill 的粒度

在进行 Skill 的创建的时候，最常遇见的一个问题就是 Skill 的粒度应该是怎样的？究竟是一个功能就是一个 Skill，还是以整个业务是一个 Skill？以 Skill 的机制、大语言模型本身的机制来看，其实不建议一个功能一个 Skill。

原因如下：

Skill 本身遵循的就是渐进式披露。如果每一个功能都是一个 Skill，那每一次启动 AI Agent 都需要加载大量的技能名称跟描述，这其实会对上下文是一种占用和浪费。而如果是一个业务一个 Skill，就能够有效减少技能数量，也能减少 Agent 的上下文占用。

当然，我也能理解有些人为什么想要一个功能就是一个 Skill，本质上是希望 Agent 能够排列组合这些 Skill，创建一些更多的业务流程。但这个其实应该是交给 CLI 来实现，把更多原子的基础功能放在 CLI 命令行中，让 Agent 通过组合命令行里的功能，实现复杂功能的编排，会比多个原子化 Skill 更好。

## Skill 的组织

### Skill.md 不应该包含全流程

虽然很多人都是以业务维度来组织 Skill，但仍然包含了大量问题。首先第一点，没有遵循渐进式披露的原则，在 Skill.md 这个文档上，就把所有业务流程跟步骤都写上去了。这会导致 Agent 读了大量使用不到的业务流程，影响它后续的执行和发挥，也会浪费很多 token。

其实正确的方式应该就像 Agents.md 一样，把 Skill.md 作为一个索引目录。在 `references` 这一个文件夹下去设计各种各样的流程，在 Skill.md 文档中说明某一个业务应该参考哪一个流程。这样 AI Agent 就不需要一次性加载所有业务流程，而是按需读取相关流程，这样能够更省 token，也能让执行更加顺畅。

### 能代码固化的不要让 Agent 执行

很多人在创建一个 Skill 时，只有一个 Skill.md 文档，没有任何协助脚本。其实很多流程是可以通过代码来固化下来的，这样能够非常有效地节省 token 消耗，并且能有效控制流程，避免在不同软件或者不同模型下执行时出现偏差，也避免了每次执行同一个 Skill 的时候模型需要重复编写代码。

还要避免一种伪固化。所谓伪固化，就是在文档中直接告诉 Agent 要执行哪一段代码，但是这一段代码需要 Agent 每次都读取、每次都自己去执行，还不如直接固化成一个脚本，让 Agent 通过调用脚本的方式完成相关指令操作。

### CLI 比 Python 脚本更有优势

能封装命令行工具会比直接提供 Python 文件更有优势。原因是命令行工具中可以提供相关的 `help`，避免模型不知道如何使用的时候去读取整个 Python 文件。如果必须读取整个 Python 文件，就达不到节省 token 的作用了。
